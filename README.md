```
                    locations/users   session cache         │
                                                    ┌───────┴────────┐
                                                    ▼                ▼
                                             [PostgreSQL]    [Firebird DB]
                                            write locations  auth · contracts
```

| Component | Description |
|-----------|-------------|
| **Bot Service** | FSM registration · Live GPS (8h) · WebApp · Branch switch · DbSessionMiddleware · WorkingTime gate |
| **API Backend** | JWT + Refresh tokens · Redis session · Firebird registry · Contract/payment endpoints · Health check |
| **Dual Database** | Async PostgreSQL (SQLAlchemy 2.0 + Alembic) · N×Firebird sync pools (dynamic registry per branch) |
| **Infra** | Docker Compose (4 services) · Webhook mode · Nginx-ready · GitHub Actions CI/CD · Pydantic v2 |

**EN:**
- Designed and built a full dual-service platform from scratch: an **aiogram Telegram bot** (field agent interface) and a **FastAPI REST backend** (dashboard API + business logic), running as separate Docker services.
- Bot service handles agent registration with FSM multi-step flow (full name → phone → branch selection), live location sharing (8-hour sessions), branch switching, and Telegram WebApp integration for a rich in-app dashboard.
- FastAPI backend exposes three router groups: `auth` (JWT + Redis token store with refresh/logout), `firebird` (contracts, payments, comments, statistics against legacy Firebird DB), and `pg` (PostgreSQL branch/location CRUD).
- Implemented **dual-database architecture**: async PostgreSQL via SQLAlchemy 2.0 (user registry, live locations, branches) + synchronous Firebird via a dynamic engine registry — each branch maps to its own isolated Firebird database file, loaded at runtime from `db.json` config.
- Built a Redis-backed JWT auth system: access token + refresh token with configurable TTL, token invalidation on logout, and user session caching to reduce DB hits on every request.
- Live GPS tracking pipeline: agents share 8-hour live location via Telegram; bot middleware validates active session from Redis, then writes GPS coordinates to PostgreSQL at configurable intervals (e.g. every 0.5 min), enabling dashboard to display real-time field agent positions.
- Firebird integration layer: dynamic engine registry pools connections to *N* Firebird databases (one per company branch); all legacy ERP queries (contracts, payment schedules, client comments, agent statistics) run through the appropriate branch DB selected by authenticated user's `workPlace` attribute.
- Swagger UI protected via HTTP Basic Auth; CORS middleware configured; Docker Compose orchestrates all services (bot, API, PostgreSQL, Redis) with health checks and dependency ordering.

**UZ:**
- Noldan to'liq ikki xizmatli platformani yaratdim: **aiogram Telegram bot** (dala agenti interfeysi) va **FastAPI REST backend** (dashboard API + biznes mantiq), alohida Docker servislari sifatida ishlaydigan.

- FastAPI backend uchta router guruhini taqdim etadi: `auth` (JWT + Redis token do'koni, refresh/logout), `firebird` (shartnomalar, to'lovlar, izohlar, statistika — Firebird DB orqali) va `pg` (PostgreSQL filial/lokatsiya CRUD).
